## Overview

AI-Assisted Dossier leverages artificial intelligence to enhance dossier management through
intelligent natural language search, automatic document summarization, question-answering over
preparation materials, and contextual content recommendations, dramatically reducing information
retrieval time during time-sensitive parliamentary sessions.

## Actors

- **Policy Officer** - Query dossiers using natural language to find specific briefing facts quickly
  during debates
- **Department Staff** - Use AI summarization to rapidly digest lengthy policy documents for
  response preparation
- **Information Manager** - Leverage automatic tagging to maintain consistent categorization across
  ministry dossiers
- **Department Coordinator** - Discover related content across multiple dossiers to ensure
  comprehensive briefings for ministry responses

## Use Cases

### UC-11.1: Natural Language Dossier Search

During live debate, department staff member types "renewable energy subsidies minister position
2024" and immediately receives relevant ministry dossier sections, even though actual documents use
different terminology like "sustainable power financial support."

### UC-11.2: Rapid Document Comprehension

Policy officer uploads 80-page environmental impact report to ministry dossier, requests AI summary,
and within seconds receives concise 500-word overview highlighting key findings, recommendations,
and action items.

### UC-11.3: Evidence-Based Question Answering

Department staff member asks "What was our department's stance on healthcare privatization last
year?" and AI provides specific answer with citations to exact dossier sections and dates for
verification.

### UC-11.4: Cross-Dossier Discovery

System automatically suggests three related dossiers from previous years when user opens current
education budget dossier, surfacing relevant historical context and precedents without manual
searching.

## Scenarios

### Scenario 1: Semantic Search During Active Debate

**Preconditions:**

- Fifty dossiers exist in education ministry with embeddings generated
- User authenticated during live parliamentary session
- Question asked about teacher salary negotiations directed at education ministry
- Traditional keyword search would miss relevant content using different terminology

**Steps:**

1. Parliamentary question asked: "What progress on educator compensation discussions?"
2. User opens search interface, types "teacher salary negotiations current status"
3. System generates vector embedding for query via LLM provider
4. System queries vector database for semantically similar content
5. LLM provider re-ranks results for relevance
6. Results displayed within 2 seconds showing three relevant sections
7. Top result from dossier titled "HR Policy Updates" (doesn't contain exact keywords)
8. User clicks result, sees highlighted section about "compensation framework discussions with
   teaching staff"
9. User finds answer: "Negotiations ongoing, proposal under minister review, expected resolution Q2
   2024"
10. Total time from query to answer: under 20 seconds

**Postconditions:**

- User found relevant information despite terminology mismatch
- Traditional keyword search would have failed
- Ministry can provide accurate, timely response
- Search query logged for quality metrics

### Scenario 2: Document Summarization Workflow

**Preconditions:**

- User has dossier editing permissions
- 65-page PDF policy document uploaded to dossier
- Document parsing libraries available
- LLM provider API accessible

**Steps:**

1. User uploads "Climate_Policy_Framework_2024.pdf" to dossier
2. System extracts plain text using document parser (15 seconds)
3. User clicks "Generate Summary" button
4. System sends document chunks to LLM provider with summarization prompt
5. Loading indicator shows "Analyzing document..."
6. After 8 seconds, 400-word summary appears
7. Summary includes: policy objectives, key stakeholders, budget implications, timeline
8. User reviews summary, confirms accuracy
9. Summary cached for 7 days for other team members
10. User adds summary to briefing notes section

**Postconditions:**

- Document summarized preserving critical information
- Team saves hours of manual reading
- Summary available for future reference
- Cache reduces repeated processing costs

### Scenario 3: Question Answering with Citations

**Preconditions:**

- Healthcare dossier contains 25 sections with generated embeddings
- User needs specific answer about policy position
- RAG (Retrieval-Augmented Generation) system operational

**Steps:**

1. User opens dossier Q&A interface
2. Types question: "What is our position on private healthcare expansion?"
3. System performs semantic search, retrieves 5 most relevant sections
4. System constructs prompt with context and question
5. Prompt includes instruction: "Answer only from provided context, cite sources"
6. LLM provider generates response over 4 seconds
7. Answer streams to UI in real-time: "The department supports limited private healthcare expansion
   in specialized care areas, while maintaining public system primacy. Private sector partnership
   focused on reducing wait times for elective procedures. (Source: Healthcare Policy Brief Section
   3.2, updated March 2024)"
8. User clicks source citation
9. System navigates to exact referenced section
10. User verifies answer against source material

**Postconditions:**

- User received grounded, verifiable answer
- Source citation enables fact-checking
- Answer based only on dossier content (no hallucinations)
- Response time acceptable for time-sensitive use

### Scenario 4: Automatic Tagging for New Dossier

**Preconditions:**

- User creating new dossier about infrastructure spending
- Dossier contains description and three sections with content
- Auto-tagging system configured

**Steps:**

1. User creates dossier titled "National Infrastructure Investment 2024"
2. Adds description: "Comprehensive overview of road, rail, and bridge funding allocations"
3. Adds sections with content about highways, public transit, structural repairs
4. Clicks "Auto-Generate Tags" button
5. System combines title, description, and section content
6. Sends to LLM provider with structured prompt requesting JSON tag array
7. LLM provider responds: `["infrastructure", "transport", "finance", "high-priority", "policy"]`
8. System parses JSON, saves tags to database
9. Tags displayed in dossier metadata
10. User reviews tags, removes "high-priority", adds "multi-year"

**Postconditions:**

- Dossier automatically categorized
- Tags enable better search and filtering
- User can refine AI suggestions
- Consistent categorization across dossiers

## Technical Overview

The AI-assisted system combines vector embeddings for semantic similarity, retrieval-augmented
generation for grounded question-answering, and large language models for summarization and tagging.
Documents converted to embeddings stored in vector database enabling sub-second similarity searches.
RAG approach ensures AI responses cite actual dossier content rather than generating unsupported
claims.

### Architecture Components

- **Embedding Generation** - Convert text to 768-dimension vectors using LLM provider embedding
  model
- **Vector Database** - PostgreSQL with pgvector extension for similarity search over embeddings
- **Document Parser** - Extract plain text from PDF, Word, Excel files for AI processing
- **RAG Pipeline** - Retrieve relevant context, augment prompts, generate grounded answers
- **Caching Layer** - Redis stores expensive AI responses to avoid regeneration

### Key Design Principles

- **Semantic Understanding** - Move beyond keyword matching to intent-based search
- **Source Grounding** - All AI responses must cite specific dossier sections for verification
- **Human Oversight** - AI suggestions reviewable and editable by users
- **Privacy First** - All processing occurs on-premise using self-hosted LLM provider
- **Performance Optimization** - Aggressive caching and asynchronous processing for responsiveness

### Integration Points

- **LLM Provider API** - Embeddings, chat completions, structured outputs for AI features
- **pgvector Database** - Cosine similarity search over document embeddings
- **MinIO Storage** - Source documents for text extraction and processing
- **Queue System** - Background jobs for embedding generation on content changes
- **Redis Cache** - Store AI responses to reduce computation costs

### Missing Components

parsing libraries not installed (smalot/pdfparser, phpoffice/phpword). Backend AI service classes
and controllers do not exist. Frontend UI components for AI features not built. Embedding generation

#### 3. Question Answering (RAG)

**Retrieval-Augmented Generation (RAG)** allows users to ask questions and get answers grounded in
dossier content.

**Use Case:**

```
User: "What was the minister's position on renewable energy subsidies?"
AI: "According to the briefing note from March 2024, the minister supports
     extending renewable energy subsidies until 2030, with a budget increase
     of 15%. (Source: Dossier #123, Section 4)"
```

**Proposed Implementation:**

**Note**: This RAG functionality does not currently exist in the codebase.

| Component                              | Purpose                                      | Key Operations                                                                                                                                                                                                                                  |
| -------------------------------------- | -------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **DossierAIService::answerQuestion()** | Answer questions using dossier content (RAG) | 1. Use semantic search to find 5 relevant sections<br/>2. Build prompt with context and question<br/>3. Send to Ollama LLM with grounding instructions<br/>4. Return answer with source citations<br/>5. Enforce "only from context" constraint |

**RAG Prompt Structure:**

- **System role**: Assistant for parliamentary debate preparation
- **Constraint**: Answer ONLY from provided context
- **Fallback**: "I don't have enough information" if answer not in context
- **Citations**: Always reference which section the answer comes from

**API Endpoint (Proposed):**

- `POST /api/dossiers/{id}/ask`
- Body: `{ "question": "What is the minister's position?" }`
- Response: `{ "answer": "...", "sources": [...] }`

**Streaming Response (Proposed):**

**Note**: Streaming functionality does not currently exist.

### Embedding Generation Pipeline

**Proposed Implementation:**

**Note**: Embedding generation pipeline does not currently exist in the codebase.

| Component                                       | Purpose                                             | Trigger                                            |
| ----------------------------------------------- | --------------------------------------------------- | -------------------------------------------------- |
| **GenerateDossierEmbeddings** (Artisan command) | Batch-generate embeddings for all existing dossiers | Manual: `php artisan dossiers:generate-embeddings` |
| **GenerateSectionEmbeddingJob** (Queue job)     | Generate embedding for single dossier section       | Automatic: when section is created or updated      |
| **DossierSectionObserver**                      | Listen for section changes and queue embedding job  | Laravel Eloquent observer                          |

**Process Flow:**

1. **Batch Processing** (Initial Setup):
   - Iterate through all dossiers and sections
   - Generate 768-dimension embedding via Ollama
   - Store in `dossier_embeddings` table
   - Progress bar for user feedback

2. **Real-time Processing** (Ongoing):
   - Section created/updated → dispatch background job
   - Job calls Ollama API to generate embedding
   - Save/update embedding in vector database
   - Enable immediate semantic search

**Performance Considerations:**

- Batch process: ~1 second per section (parallelize with queues)
- Real-time: < 2 seconds for single embedding
- Queue system prevents blocking user requests

## Known Limitations

### 1. Context Window Size

**Issue:** LLMs have limited context (8K-128K tokens)

**Impact:** Can't process entire dossiers in one go

**Workaround:** Chunking and retrieval (RAG approach)

### 2. Hallucinations

**Issue:** LLMs sometimes generate plausible but false information

**Mitigation:**

- Always require citation of sources
- Implement confidence scores
- Human review for critical content

### 3. Language Support

**Issue:** Ollama models may not be optimized for Dutch

**Workaround:**

- Use multilingual models
- Fine-tune on Dutch government documents (future)

### 4. Cold Start Time

**Issue:** First Ollama request can take 10+ seconds (model loading)

**Mitigation:** Keep models warm with periodic health checks

## Testing Considerations

### Test Scenarios

1. **Semantic Search Accuracy**: Measure precision/recall against human-labeled queries
2. **Summarization Quality**: Compare AI summaries to human-written abstracts
3. **Question Answering**: Verify answers are grounded in source documents
4. **Auto-tagging**: Measure agreement with human-assigned tags
5. **Load Testing**: Concurrent AI requests don't degrade performance

### Evaluation Metrics

- **Search Relevance**: NDCG@10 (Normalized Discounted Cumulative Gain)
- **Summarization**: ROUGE score vs human summaries
- **QA Accuracy**: Exact match and F1 score
- **Tagging**: Precision, recall, F1 for each tag category
- **Latency**: p50, p95, p99 response times
